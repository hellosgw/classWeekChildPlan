## 序言

这个项目始于一个简单的需求：**幼儿园发了一张 PDF 周计划，家长需要一份可执行的家庭同步方案。**

不是 AI 玩具。不是技术演示。是一个每天真实陪伴 4 岁孩子的工具——18:30 的自主探索，20:00 的深度连接。每一份生成的计划，都会在真实的亲子时间里被执行。

我们从零开始，没有 Python 环境，没有工作流引擎，只有一份 PDF 和一个目标。经过连续的 Engineer Loop，我们建立了从环境安装到计划存档的完整管道。这个项目的每一行代码都来自真实痛点，每一个改进都来自实际运行中的发现。

**本宪章不是束缚，而是承诺：** 承诺用户数据不会泄露，承诺每次生成的计划都不会丢失，承诺每次迭代都验证通过才交付，承诺这个项目始终保持"一张 PDF → 一份可执行的计划"的清晰边界。

## 第I条 — 项目身份

### 1.1 定义
`classWeekChildPlan` 是一个**轻量级、单用途**的家园共育工具。它严格做一件事：把幼儿园周活动计划 PDF 转化为家庭同步执行计划。

### 1.2 不是
- 不是通用 AI 平台
- 不是内容管理系统
- 不是多人协作工具
- 不是移动端 App

### 1.3 核心主张

| 原则 | 含义 |
|:---|:---|
| 单文件优先 | 前端单页 HTML，无构建工具链 |
| 零运行时依赖 | 除了 Python + pip 包，不依赖数据库/容器/云服务 |
| 可见即所得 | 生成的计划立即可预览、可保存、可回看 |
| 数据归用户 | PDF 和生成的计划存储在本地，不发送到第三方 |

## 第II条 — 数据主权

### 2.1 API Key
`DEEPSEEK_API_KEY` 是唯一的密钥。它只用于与 DeepSeek API 通信，不会被记录、缓存或发送到其他端点。未配置 Key 时，系统拒绝服务并给出明确提示。

### 2.2 PDF 文件
上传的 PDF 仅在服务端内存中完成文本提取，原始文件不会以任何形式长期存储。提取的文本仅用于本次 AI 调用，调用完成后不保留。

### 2.3 生成的计划
每次 AI 生成的家庭同步计划**必须自动持久化**，确保用户不会因为页面刷新丢失内容。持久化采用双轨制：

- **前端**：localStorage 即时保存，刷新可回看
- **后端**：`.whale/output/weekly_plans/` 文件归档，跨会话不丢失

### 2.4 不可绕过
任何对 `generate_plan` 流程的修改不得移除持久化步骤。前端的 `savePlanToHistory()` 调用和后端的 `_save_plan()` 调用是返回响应前的必经环节。

## 第III条 — 质量纪律

### 3.1 PLAN 前置
任何修改开始前，必须先执行 PLAN。PLAN 包含：
- 当前状态检查
- 目标确认
- 风险识别

未经过 PLAN 阶段的修改不得执行。

### 3.2 DISPATCH 原子性
每个 DISPATCH 只做一件事。多个独立修改应并行执行，但单个操作必须是原子的。

### 3.3 VERIFY 强制
每个 DISPATCH 之后必须有对应的 VERIFY。VERIFY 的标准是：

| 改动类型 | 验证方式 |
|:---|:---|
| 后端 API 新增 | 返回 HTTP 200 + 正确的 JSON 结构 |
| 后端逻辑修改 | 已有 PDF 上传能正常生成计划 |
| 前端 UI 改动 | 上传 → 渲染 → 保存 → 回看链路完整 |
| 持久化相关 | 刷新页面后历史记录可读 |
| 脚本/自动化 | 完整执行不报错，输出报告有效 |

**没有 VERIFY 的 DISPATCH 视为未完成。**

### 3.4 RECORD 义务
每次修改完成后，必须记录：
- 改了什么（文件/行号/变更说明）
- 为什么改（对应的用户痛点是哪个）
- 本次发现了什么新问题（LOOP 输入）

记录写入 `.whale/LOOP_ENGINEER_HANDBOOK.md` 或对应章节。

## 第IV条 — 迭代伦理

### 4.1 Loop 循环
本项目的所有改进遵循 **PLAN → DISPATCH → VERIFY → RECORD → LOOP** 五步循环。每一次循环都让系统比上一轮更健壮。

### 4.2 一次一个问题
迭代以用户真实痛点为驱动。每次 Loop 处理一个完整的问题——从发现到验证到沉淀。不允许在一次 Loop 中同时处理两个不相关的问题。

### 4.3 经验必须固化
在 Loop 中发现的问题（如 Python 环境缺失、PowerShell 版本兼容性、文件未归档）必须在同一个 Loop 中写入经验文档或脚本注释。**问题发现地和问题修复地之间不能有间隙。**

### 4.4 向后兼容
任何修改不得破坏已有功能的核心链路：PDF 上传 → AI 生成 → 卡片渲染 → 保存回看。如果修改必须改变链路行为，需要在 AGENTS.md 中更新架构说明。

## 第V条 — 协作规则

### 5.1 与 AI 的交互
当开发者与 AI 协作时：
- AI 使用 Loop Engineer 框架，而非直接输出方案
- 每次修改前展示 PLAN，获得确认后再执行
- 修改完成后提供 VERIFY 证据
- 经验必须沉淀到项目文件中

### 5.2 工具使用纪律

| 场景 | 正确做法 | 原因 |
|:---|:---|:---|
| 长时命令 | `task_shell_start` 而非 `exec_shell` | 后台进程跨回合存活 |
| 文件上传 | `curl.exe` 而非 `Invoke-WebRequest -Form` | PS5.1 无 -Form 参数 |
| Python 定位 | `$env:LOCALAPPDATA\...\python.exe` 候选路径 | WindowsApps stub 是 0 字节占位符 |
| 后台日志 | stdout/stderr 分两个文件 | `Start-Process` 不支持同一文件 |

### 5.3 错误透明
所有失败的 DISPATCH 必须报告明确的错误原因，不允许静默失败。错误信息应包含：失败步骤、错误代码/信息、建议的修复方向。

## 第VII条 — 法律层级

当指令冲突时，优先级如下：

```
 1.  本宪章（第I-VI条）              不可违背
 2.  用户当前指令                     在宪法范围内最高
 3.  AGENTS.md                        项目级规则
 4.  工具输出证据                     高于任何推测
 5.  经验手册记录                     >
 6.  代码注释                         >
 7.  惯例（如 `feat:` 前缀）          最低优先级
```

**示例：**

- **用户说"不要保存计划了"** → 违背第II条第2.3款 → 解释边界并提供替代方案
- **用户说"帮我加个新功能"** → 先执行第III条第3.1款 PLAN → 再 DISPATCH
- **经验手册说"用 curl.exe"而代码注释写"Invoke-WebRequest"** → 经验手册优先（来自实际运行教训）
